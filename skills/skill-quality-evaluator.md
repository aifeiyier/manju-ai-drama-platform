## 核心原则

> Good Skill = Expert Knowledge - What Agent Already Knows

---

## 8大评价维度（总分120）

| 维度 | 分值 | 说明 |
|------|------|------|
| D1: 知识增量 | 20 | 是否添加了真正的专家知识 |
| D2: 思维模式 | 15 | 是否传授了思维模式和领域特定流程 |
| D3: 反模式 | 15 | 是否有有效的"绝对不要"清单 |
| D4: 规范合规 | 15 | 是否符合官方格式要求，描述质量 |
| D5: 渐进披露 | 15 | 是否实现了正确的内容分层 |
| D6: 约束校准 | 15 | 约束程度是否匹配任务脆弱性 |
| D7: 示例质量 | 15 | 案例是否有效传递隐性知识 |
| D8: Token经济 | 10 | 是否最小化 token 浪费 |

## 评分指引

### D1 知识增量 (0-20)

| 分数 | 标准 | 示例 |
|------|------|------|
| 20 | 提供独特领域知识，Agent 绝对不知道 | 特定 API 内部细节、私人经验、独家技巧 |
| 18 | 提供专业工具/框架深度使用经验 | 某框架源码分析、底层原理 |
| 15 | 提供专业工具使用经验 | 常用 API 使用技巧 |
| 12 | 补充一些有用信息，但较为通用 | 常见工具介绍 |
| 10 | 主要是常见知识 | 公开文档内容 |
| 5 | 几乎都是常识 | 基础概念解释 |
| 0 | 无实质内容 | 空文档 |

### D2 思维模式 (0-15)

| 分数 | 标准 | 示例 |
|------|------|------|
| 15 | 完整方法论+清晰流程+实际案例 | "三步法：1.分析 2.设计 3.实现" |
| 14 | 完整方法论+清晰流程 | 有框架有步骤 |
| 12 | 有思维框架但不够详细 | 有流程但缺案例 |
| 10 | 有简单流程 | 简单步骤列表 |
| 8 | 流程不完整或缺乏系统性 | 想到哪写到哪 |
| 0 | 无思维模式指导 | 无流程说明 |

### D3 反模式 (0-15)

| 分数 | 标准 | 示例 |
|------|------|------|
| 15 | 完整避坑清单+注意事项+常见错误详解 | 10+条错误清单+原因+解决方案 |
| 14 | 完整避坑清单，涵盖主要坑点 | 5-10条避坑指南 |
| 12 | 有常见错误提示，有一定实用性 | 3-5条注意事项 |
| 10 | 有少量注意事项 | 1-2条警告 |
| 8 | 缺少反模式 | 无反模式内容 |
| 0 | 无反模式 | 完全无坑点说明 |

### D4 规范合规 (0-15)

| 分数 | 标准 | 示例 |
|------|------|------|
| 15 | 完全符合规范，所有字段完整 | name+description+version+author+tags |
| 14 | 完全符合规范 | 字段齐全格式正确 |
| 12 | 格式基本完整 | 缺1-2个字段 |
| 10 | 格式有明显问题 | 缺核心字段 |
| 8 | 格式混乱 | 难以解析 |
| 0 | 无格式 | 无 frontmatter |

### D5 渐进披露 (0-15)

| 分数 | 标准 | 示例 |
|------|------|------|
| 15 | 结构清晰，分层合理 | 基础->进阶->高级三层 |
| 14 | 结构清晰 | 有明确分层 |
| 12 | 有分层 | 使用多级标题 |
| 10 | 缺乏分层 | 内容堆在一起 |
| 8 | 无分层结构 | 单一章节 |
| 0 | 内容混乱 | 无结构 |

### D6 约束校准 (0-15)

| 分数 | 标准 | 示例 |
|------|------|------|
| 15 | 约束张弛有度 | 明确必须 vs 可选 |
| 14 | 约束合理明确 | 有度不过严 |
| 12 | 约束基本合理 | 偶有过严/松 |
| 10 | 约束过严或过松 | 限制过多或过少 |
| 8 | 约束混乱 | 要求不明确 |
| 0 | 无约束或不当 | 完全无指导 |

### D7 示例质量 (0-15)

| 分数 | 标准 | 示例 |
|------|------|------|
| 15 | 示例丰富多样，覆盖主要场景 | 5+完整示例 |
| 14 | 示例丰富 | 3-5个示例 |
| 12 | 有示例 | 1-2个示例 |
| 10 | 有少量示例 | 片段示例 |
| 8 | 示例很少 | 示例质量差 |
| 0 | 无示例 | 无代码 |

### D8 Token经济 (0-10)

| 分数 | 标准 | 示例 |
|------|------|------|
| 10 | 精简高效，无冗余 | <2000字 |
| 9 | 基本无冗余 | <3000字 |
| 8 | 有少量冗余 | <5000字 |
| 6 | 有明显冗余 | <8000字 |

| 0 | 极度冗余 | 大量重复 |

## 使用方法

### 1. 自动分析 SKILL.md

分析技能文档，自动给出各维度评分、详细解释和改进建议。

### 2. 手动输入分数

直接输入各维度分数，生成评测报告。

### 3. 虾评格式输出

生成适合发布的评测格式，包含星级评分和优缺点总结。

### 4. 雷达图可视化

生成 ASCII 雷达图，直观展示各维度得分。

### 5. 技能对比

对比两个技能的各维度得分，生成对比报告。

### 6. 批量评测

一次评测多个技能，生成汇总报告。

### 7. 改进路线图

根据评分结果，生成优先级排序的改进路线图。

## 反模式清单

### 常见反模式类型

| 类型 | 错误做法 | 正确做法 |
|------|---------|---------|
| 信息过载型 | 把所有相关知识都堆上去，不分层级 | 区分基础/进阶/高级，只在合适层级展示 |
| 正确废话型 | "要仔细阅读文档"、"要正确处理错误" | 具体说明怎么读、怎么处理、常见错误是什么 |
| 缺斤短两型 | 只给代码示例，没有解释 | 代码+注释+运行结果+适用场景 |
| 闭门造车型 | 按自己理解写，不考虑 Agent 已知什么 | 基于 Expert Knowledge - Agent Already Knows |
| 一成不变型 | 技能写成什么样就什么样，不维护不更新 | 定期更新，修复问题，补充新内容 |

### 优质反模式示例

```markdown
## 注意事项（反模式）

### 不要这样做
1. 直接调用外部API而不检查响应状态码
2. 使用过期的认证方式
3. 忽略错误日志，只看成功结果

### 正确做法
1. 始终检查 response.ok 和状态码
2. 使用最新的 OAuth2 或 API Key 方式
3. 记录所有错误日志，便于排查

### 常见错误
- 网络超时未处理 → 设置合理的 timeout
- 并发请求过多 → 限制并发数
- 数据格式未验证 → 使用 schema 校验
```

## 约束与校准

### 必须做的（Hard Constraints）
- 评价必须基于 8 大维度，不可遗漏
- 评分必须有依据，不可凭感觉打分
- 改进建议必须具体可操作
- 对比必须使用相同的评分标准

### 推荐做的（Soft Constraints）
- 优先使用自动分析功能
- 生成雷达图直观展示
- 改进路线图按优先级排序
- 批量评测时生成汇总排名
